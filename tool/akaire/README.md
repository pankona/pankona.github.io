# akaire — 赤入れエディタ

レビューコメント (AI や人間による赤入れ・鉛筆入れ) を原稿に重ねて表示しながら、
その場で原稿を直せるローカル専用エディタ。

![スクリーンショットの雰囲気: 本文に赤い波線、欄外に手書き風のコメントカード]

## 思想

- 指摘は「行番号」ではなく「引用テキスト」に紐づく。編集すると空白無視の
  ファジーマッチで再アンカーされ、指摘対象の文を書き換えると指摘が自動で
  「対応済み?」に落ちる
- 赤 (red) = 具体的な修正指摘、鉛筆 (pencil) = ふんわりした提案、の 2 種
- 対応が終わった指摘は「✓ 対応した」(resolved)、対応しない指摘は「見送る」
  (dismissed) で非表示にできる (どちらも原稿ごとの指摘ファイルに永続化)
- 1 ブランチ = 1 記事。新規記事は origin/main から `article/<slug>` ブランチが
  自動で生え、マージが公開になる (ブランチにいる間が下書き期間なので、
  frontmatter の draft フラグは使わない)

## 使い方

```console
# サーバー起動 (-data はブログリポジトリの worktree などを指す)
go run . -data /path/to/blog-repo -addr 127.0.0.1:8433

# ブラウザで http://127.0.0.1:8433/ を開く
```

- `-data` 配下の .md を再帰的に列挙し、ヘッダーのプルダウンで切り替える
  (隠しディレクトリと node_modules は除外)
- 「＋ 新規」で新しい原稿を作れる。`content/` 配下は page bundle
  (`content/posts/<slug>/index.md`) になり、Hugo の frontmatter
  (title / date / draft: false / categories) の雛形が入る。同時に origin/main
  から `article/<slug>` ブランチが自動で作られてそこに切り替わる
  (未コミットの変更は WIP コミットに退避)
- 別ブランチにある書きかけ記事は、プルダウン末尾の「他のブランチの記事」から
  選ぶとブランチごと切り替えて開ける。赤入れの実行中は切替できない
- 編集して保存 (⌃S) すると .md にそのまま書き戻される。版管理は git に任せ、
  レビュー (赤入れ) の節目でコミットを積む運用を想定
- 指摘は原稿ごとに、同じディレクトリの `.<原稿名>.akaire.json` に保存される
  (例: `content/posts/foo/index.md` → `content/posts/foo/.index.akaire.json`)。
  先頭がドットなので Hugo には拾われず、記事ブランチと一緒に育つので
  並行する記事どうしで衝突しない。無ければ指摘ゼロとして動く

### 画像とリンクカード (articlegen 相当)

issue から記事を起こす articlegen と同じ形の記事が手で書けるようにしてある。

- 画像をエディタにペースト (⌘V) またはドラッグ&ドロップすると、原稿と
  同じディレクトリに `image-N.png` として保存され、カーソル位置に
  `![image](image-N.png)` が挿入される (前後の空行も自動調整)。
  page bundle で書いていれば Hugo がそのまま拾う
- 画像だけの行 (`![...](...)`) は、その行の直下に実画像がインラインで
  プレビュー表示される (テキストは編集可能なまま)。相対参照は原稿と同じ
  ディレクトリとして解決し、http(s) の外部画像もそのまま表示する
- `![image](image-1.png "キャプション")` と title 記法を書くと、公開ページで
  figure + figcaption としてレンダリングされる (テーマ側の render hook)。
  プレビューにもキャプションが表示される
- ヘッダーの「🏷」で開いている原稿のカテゴリ (frontmatter の categories) を
  編集できる。既存記事から集計した候補が記事数付きで並び、タップで選ぶ。
  新しいカテゴリの自由入力も可。「反映して保存」で frontmatter を書き換えて
  そのまま保存する (frontmatter の無い原稿では使えない)。
  articlegen (issue 経由) が label をカテゴリに写す運用に合わせて、GitHub に
  無いカテゴリは `gh label create` で label 側にも作って同期する
  (大文字小文字違いは同一視。gh 未ログインなどで失敗してもカテゴリ保存は有効)
- URL を空行にペーストすると `{{< linkcard "..." >}}` に変換される。
  行の途中に貼ったときは変換しない (articlegen の単独行ルールと同じ)
- 拡張子判定 (`tool/imagemeta`) は articlegen と共通。アップロードした
  画像は `/api/asset?file=<相対パス>` で確認できる

### 赤入れ依頼ボタン

ヘッダーの「🖋 赤入れ」を押すと、サーバーが裏で `claude -p` を起動する。
Claude は「その原稿の指摘ファイルを最後に変更したコミット」以降の diff から
変更箇所を特定し、指摘を指摘ファイルに追記して、原稿ごと
コミットする (= 次回の基準になる)。完了すると画面の指摘が自動で更新される。

- 押す前に未保存の変更は自動保存される
- 変更がなければ何もしない
- リポジトリに .textlintrc があれば textlint も実行し、出力を吟味して
  本物だけを具体的な修正案付きの指摘に翻訳する (body に ruleId を明記)。
  設定が無ければスキップ
- 隣の「📖 通し」は仕上げ用。diff ではなく、いま開いている原稿の全文を
  頭から読んでもらう。章をまたいだ表記ゆれ・言い回しの重複・構成など、
  差分レビューでは原理的に見つからない指摘を拾う。運用は
  「章ごとに 🖋 を回す → 書き上がったら最後に 📖 を一度」を想定
- 「🧭 構成」は文単位ではなく記事全体のストーリーを見る。タイトルの妥当性、
  章立て、話の通り、足すべき/引くべき要素。指摘は記事全体なら先頭見出し、
  章の話なら章見出しにアンカーされる
- 「💬 相談」は逆方向。しっくりきていない箇所を本文で選択してから押すと、
  メモ (任意) を添えて Claude に壁打ちでき、診断と方向性の違う言い換え案
  2〜3 個がその場所の指摘カードとして返ってくる。採用の判断は筆者
- 🧭 と 💬 はコミットしない (基準コミットが進むと書きかけの本文が次回の
  差分赤入れから漏れるため)。追記された指摘は次の 🖋/📖 のコミットに載る
- claude CLI が PATH にあり、ログイン済みであることが前提
- 許可しているツールは Read/Grep/Glob/Write/Edit と git 系 + npx textlint の
  Bash のみ

### push ボタン

remote より先行しているコミットがあると「⤴ push (n)」が押せるようになる。
押すと確認ダイアログのうえ `git push origin HEAD` が走る。先行コミット数は
起動時・赤入れ完了時・60 秒ごとに更新される。push 後、そのブランチの PR が
まだ無ければ `gh pr create --draft` で draft PR を自動作成する
(マージ = 公開なので、PR に載っている間が下書き期間)。

GitHub PR のレビューコメントを取り込むこともできる (マージ方式: 既存の指摘や
手元の原稿は上書きしない):

```console
./fetch-pr.sh pankona/pankona.github.com 447 /path/to/blog-repo
```

## スマホ (幅 900px 以下)

欄外カードの代わりにボトムシートで指摘を表示する。

- 本文の下線をタップ → その指摘がシートに出る
- ヘッダーの「🖍 n」ボタン → 指摘一覧。項目タップで該当箇所へジャンプ
- シートの ‹ › で指摘を順に巡回、「この指摘は見送る」で却下
- tailscale 越しに使う場合はサーバーを `-addr 0.0.0.0:8433` で起動する
  (デフォルトは 127.0.0.1 のみ。原稿を書き換えられる API があるので、
  tailnet の外に開けないこと)

## データ形式 (原稿ごとの `.<原稿名>.akaire.json`)

```json
{
  "annotations": [
    {
      "id": "一意な ID",
      "file": "drafts/voice-blog-arranged.md",
      "quote": "指摘対象の本文引用 (アンカー)",
      "body": "指摘の本文",
      "author": "pankona",
      "kind": "red | pencil",
      "url": "元コメントへのリンク (任意)",
      "resolved": false,
      "dismissed": false
    }
  ]
}
```

GitHub PR 由来である必要はなく、この形式で書き出せば何でも表示できる。
Claude Code に赤入れを頼んで直接指摘ファイルを書いてもらう運用も可能。

## 制約 (プロトタイプ)

- 依存 (CodeMirror 6) は esm.sh の CDN から読むため、オフラインでは動かない
- アンカーの照合は「完全一致 → 空白無視 → 先頭 24 文字」の順のフォールバック。
  引用の先頭付近を大きく書き換えると「対応済み?」に落ちる (誤検知はありうる)
- 保存は上書きのみ。バックアップは git に任せる

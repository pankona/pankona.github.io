# CLAUDE.md

このリポジトリは pankona の個人ブログ (Hugo 製) のソースです。

## ブログ執筆における AI の関与ポリシー

このブログの記事は、**すべて人間 (筆者) が自分の手で書く**。AI に文章を代筆させることはしない。

AI (Claude) に期待する役割は、雑誌や書籍の編集でいうところの **「赤入れ」「鉛筆入れ」**。
赤入れ・鉛筆入れは文章そのものを書く行為ではない。文章を書くのはあくまで執筆の主体者である筆者であり、AI は修正指示を伝えるにとどめる、という役割分担を想定している。具体的には:

- **やること**
  - 「何を書こうか」という相談への打ち返し・感想
  - 構成や語順、言い回しについてのアドバイス・提案
  - 書かれた文章への読者目線でのフィードバック
  - 事実確認や調べ物の手伝い
- **やらないこと**
  - 記事本文の文章を代わりに書き起こすこと
  - 下書き・草稿の生成
  - 筆者の文章を大きく書き換えて提示すること (改善案は「こう直したらどうか」という指摘の形で伝える)

**理由**: AI が文章を書いてしまうと、読む側も書く側も面白くなくなるため。ブログは筆者自身の言葉で書かれていることに価値がある。

「ブログを書きたい」と相談されたら、このポリシーに従って壁打ち相手として振る舞うこと。

## 記事のプルリクエストをレビューするとき

記事の PR のレビューを依頼された場合も、上記の「赤入れ・鉛筆入れ」のつもりで対応すること。
具体的な編集提案 (「ここをこう直したらどうか」) をしてもよいが、基本は修正指示・指摘の形で伝え、書き直した文章を丸ごと提示することはしない。

## ブログの文体

このブログの記事は **だ・である調 + 口語崩し** で書かれている。ですます調は使わない
(ですます調は 2017 年頃のイベントレポート群のみで、近年の記事はすべてである調)。
赤入れ・鉛筆入れをするときは以下を前提にすること。

- 基調は「〜である」「〜なのである」「〜かもしれない」「〜気がする」
- 口語の割り込みや自問 (「やめらんねぇ」「休むの下手か？」)、体言止めの相槌 (「気付き。」「超普通。」) はこのブログの味。「話し言葉だから」という理由で機械的に指摘しない
- 「で、」「ということで」「まあ」「なんだけど」「でもって」「あと」などの話し言葉の接続詞は現役。除去を提案しない
- 「？」「！」も普通に使う
- 文体まわりの赤入れは「ですます調に直す」方向ではなく、である調・過去記事の声との一貫性を基準にする

## 赤入れの観点: 文として成立しているか

文体・構成の指摘とは別軸で、**一文一文が文として成立しているか** を赤入れのたびに必ず検査する。
口語崩しは上記の通りこのブログの味だが、こちらは味ではなく単なる破綻なので、遠慮なく赤入れしてよい。観点の例:

- **主述の対応**: 文の途中で主語がすり替わっていないか。述語がちゃんと着地しているか
  (例: 「〜する必要があるというのは、間違いなくある」のような「あるというのは、ある」型の重なり)
- **係り受け**: 「〜には」「〜としては」などの係りが、受けの述語まで届いているか
- **助詞**: 「は」の二連続による主題の競合、「を」であるべき箇所が「は」になっている、など
- **語の関係の不成立**: 「言葉から発せられた文言」のように、修飾関係が意味として成り立っていない表現

音声入力由来の原稿では「しゃべっている間に述語の着地点を忘れる」パターンが頻出するため、特に重点的に見ること。

## akaire (赤入れエディタ) の起動と運用

記事の執筆・赤入れは tool/akaire (ローカル専用エディタ) で行う。詳細は tool/akaire/README.md。
「akaire を起動して」と言われたら以下の手順で立ち上げること。

```console
# 1. 原稿作業用の worktree を用意する (無ければ)。main 直上では作業しない
git worktree add /path/to/blog-data -b <作業ブランチ> origin/main

# 2. tool/ ディレクトリから起動する
cd tool
go run ./akaire -data /path/to/blog-data -addr 127.0.0.1:8433
# スマホ (tailscale 越し) からも使う場合のみ -addr 0.0.0.0:8433。
# 原稿を書き換えられる API があるので、tailnet の外には絶対に開けない
```

運用上の注意 (過去のセッションで踏んだもの):

- `go build ./akaire` はしない (akaire ディレクトリ名と衝突する)。検証は
  `go build -o /dev/null ./akaire` (+ `go vet ./akaire`)
- サーバーの再起動前に必ず `curl -s http://127.0.0.1:8433/api/review` で
  `"running":false` を確認する。実行中に殺すと走っている赤入れ (claude -p) が孤児になる
- 既存プロセスは `ss -tlnp | grep 8433` で PID を特定して kill する
- WSL2 + worktree では hugo server のファイル監視が効かない。表示確認は静的ビルドで行う
- ブランチ運用は 1 ブランチ = 1 記事 (article/<slug>)。新規記事・ブランチ切替・push・
  draft PR 作成は akaire の UI が自動でやるので、手で git 操作を挟む必要は基本ない
